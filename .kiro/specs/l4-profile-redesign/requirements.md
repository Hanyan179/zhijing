# Requirements Document

## Introduction

本文档定义 L4 核心知识层的重新设计需求。核心任务是：
1. **分析对比**：三层维度架构 vs 原内核状态设计的优劣
2. **验证完整性**：确保前两层维度完整覆盖人生各方面
3. **重构现有设计**：基于分析结果更新 L4-PROFILE-EXPANSION-PLAN.md

**7个一级维度体系**：
- 5个核心维度：本体、物质、成就、阅历、精神
- 关系维度：独立子系统（本次暂不实现，但需预留接口）
- AI偏好维度：独立子系统（本次暂不实现，但需预留接口）

**设计哲学**：
- 第1层 & 第2层：标准化、固定，涵盖"正常人的大多数"情况
- 第3层：预设通用细分领域，由 AI 动态维护、新增或更新
- 现有中间表（SourceLink、ExtractedSourceLink）需重构以支持多对多关联

## Glossary

- **L4_Layer**: 核心知识层，存储高度抽象的画像、标签节点与事实
- **Life_OS**: 人生操作系统，用户画像的维度体系框架
- **Dimension_Level_1**: 一级维度，7个顶层分类（本体、物质、成就、阅历、精神、关系、AI偏好）
- **Dimension_Level_2**: 二级维度，固定的子分类（如身份认同、身体状态等）
- **Dimension_Level_3**: 三级维度，AI动态维护的细分领域
- **KnowledgeNode**: 知识节点，存储具体维度数据的通用结构
- **SourceLink**: 溯源链接，连接 L4 知识节点与 L1 原始数据（现有实现）
- **ExtractedSourceLink**: 提取溯源链接，API 返回的简化版本（现有实现）
- **Confidence**: 置信度，AI 提取信息的可信程度 (0.0~1.0)
- **StaticCore**: 静态内核，用户手动输入的基础信息（现有实现）

## Requirements

### Requirement 1: 架构方案分析与对比

**User Story:** As a 系统架构师, I want 分析三层维度架构与原内核状态设计的优劣, so that 能做出最佳设计决策。

#### Acceptance Criteria

1. THE Design_Document SHALL include a comparison analysis of "三层维度架构" vs "原内核状态设计"
2. THE Analysis SHALL cover: 数据组织方式、扩展性、AI维护难度、用户理解成本、实现复杂度
3. THE Analysis SHALL provide clear recommendations with rationale
4. THE Design_Document SHALL document the final decision and reasoning

### Requirement 2: 维度完整性验证

**User Story:** As a 系统架构师, I want 验证前两层维度是否完整覆盖人生各方面, so that AI能基于此进行扩展。

#### Acceptance Criteria

1. THE Design_Document SHALL verify that Level_1 and Level_2 dimensions cover all major life aspects
2. THE Verification SHALL check for: 遗漏的重要生活领域、维度之间的逻辑冲突、维度归属的合理性
3. IF any gaps are found, THEN THE Design_Document SHALL propose additions or adjustments
4. THE Design_Document SHALL confirm that Level_3 can be dynamically generated by AI based on Level_1 and Level_2

### Requirement 3: 七大一级维度定义

**User Story:** As a 系统架构师, I want 定义完整的七大一级维度体系, so that 系统有清晰的顶层结构。

#### Acceptance Criteria

1. THE Life_OS_System SHALL define exactly 7 first-level dimensions: 本体(Self), 物质(Material), 成就(Achievements), 阅历(Experiences), 精神(Spirit), 关系(Relationships), AI偏好(AI_Preferences)
2. THE System SHALL mark 关系(Relationships) as "独立子系统-暂不实现" with interface reserved
3. THE System SHALL mark AI偏好(AI_Preferences) as "独立子系统-暂不实现" with interface reserved
4. THE System SHALL implement the 5 core dimensions (本体、物质、成就、阅历、精神) in this phase

### Requirement 4: 本体维度 (The Self) - 完整三层结构

**User Story:** As a 用户, I want 记录我的身份认同、身体状态和性格特质, so that AI 能理解我的核心自我。

#### Acceptance Criteria

1. THE Self_Dimension SHALL contain exactly 3 second-level dimensions: 身份认同(Identity), 身体状态(Physical), 性格特质(Personality)
2. THE Identity_Dimension SHALL contain third-level items: 社会角色(Social_Roles), 职业身份(Professional_Identity), 外在形象(Appearance_Style)
3. THE Physical_Dimension SHALL contain third-level items: 健康状况(Health_Condition), 睡眠质量(Sleep_Quality), 饮食习惯(Dietary_Habits)
4. THE Personality_Dimension SHALL contain third-level items: 自我评价(Self_Assessment), 行为偏好(Behavioral_Preferences)
5. THE System SHALL allow AI to add additional third-level items such as: 潜意识模式, 思维习惯

### Requirement 5: 物质维度 (Material) - 完整三层结构

**User Story:** As a 用户, I want 记录我的经济状况、物品环境和生活保障, so that AI 能理解我的物质世界。

#### Acceptance Criteria

1. THE Material_Dimension SHALL contain exactly 3 second-level dimensions: 经济状况(Economy), 物品与环境(Objects_Space), 生活保障(Security)
2. THE Economy_Dimension SHALL contain third-level items: 资产概况(Asset_Status), 消费行为(Consumption), 负债压力(Debt_Pressure)
3. THE Objects_Space_Dimension SHALL contain third-level items: 常用物品(Possessions), 居住环境(Living_Environment), 收藏爱好(Collections)
4. THE Security_Dimension SHALL contain third-level items: 保险与风控(Insurance_Safety)

### Requirement 6: 成就维度 (Achievements) - 完整三层结构

**User Story:** As a 用户, I want 记录我的事业发展、个人能力和成果展示, so that AI 能理解我的成就。

#### Acceptance Criteria

1. THE Achievements_Dimension SHALL contain exactly 3 second-level dimensions: 事业发展(Career), 个人能力(Competencies), 成果展示(Outcomes)
2. THE Career_Dimension SHALL contain third-level items: 工作经历(Work_Experience), 商业活动(Business_Activities)
3. THE Competencies_Dimension SHALL contain third-level items: 专业技能(Professional_Skills), 学习进修(Education_Learning), 生活才艺(Life_Talents)
4. THE Outcomes_Dimension SHALL contain third-level items: 个人作品(Personal_Creations), 荣誉认可(Recognition_Awards)

### Requirement 7: 阅历维度 (Experiences) - 完整三层结构

**User Story:** As a 用户, I want 记录我的文化娱乐、探索足迹和人生历程, so that AI 能理解我的经历。

#### Acceptance Criteria

1. THE Experiences_Dimension SHALL contain exactly 3 second-level dimensions: 文化娱乐(Culture_Entertainment), 探索足迹(Exploration), 人生历程(History)
2. THE Culture_Entertainment_Dimension SHALL contain third-level items: 阅读体验(Reading), 视听欣赏(Movies_Music), 游戏娱乐(Gaming)
3. THE Exploration_Dimension SHALL contain third-level items: 旅行见闻(Travel_Stories), 探店体验(Lifestyle_Exploration)
4. THE History_Dimension SHALL contain third-level items: 重要节点(Milestones), 往事回忆(Memories)

### Requirement 8: 精神维度 (Spirit) - 完整三层结构

**User Story:** As a 用户, I want 记录我的意识形态、心理状态和思考感悟, so that AI 能理解我的精神世界。

#### Acceptance Criteria

1. THE Spirit_Dimension SHALL contain exactly 3 second-level dimensions: 意识形态(Ideology), 心理状态(Mental_State), 思考感悟(Wisdom)
2. THE Ideology_Dimension SHALL contain third-level items: 价值观(Values), 愿景梦想(Visions_Dreams)
3. THE Mental_State_Dimension SHALL contain third-level items: 情绪感受(Emotions), 压力来源(Stressors)
4. THE Wisdom_Dimension SHALL contain third-level items: 观点看法(Opinions), 反思复盘(Reflections)

### Requirement 9: 关系维度预留接口

**User Story:** As a 系统架构师, I want 为关系维度预留接口, so that 后续能独立实现关系子系统。

#### Acceptance Criteria

1. THE System SHALL define Relationships as a first-level dimension with status "reserved"
2. THE System SHALL maintain existing NarrativeRelationship model as the foundation
3. THE System SHALL ensure KnowledgeNode can reference relationship entities via relationshipId
4. THE System SHALL support entity reference format: [REL_ID:displayName]
5. THE Design_Document SHALL document the planned relationship subsystem interface

### Requirement 10: AI偏好维度预留接口

**User Story:** As a 系统架构师, I want 为AI偏好维度预留接口, so that 后续能独立实现AI偏好子系统。

#### Acceptance Criteria

1. THE System SHALL define AI_Preferences as a first-level dimension with status "reserved"
2. THE System SHALL maintain existing AIPreferences model as the foundation
3. THE Design_Document SHALL document the planned AI preferences subsystem interface

### Requirement 11: 现有中间表重构

**User Story:** As a 开发者, I want 重构现有的SourceLink和ExtractedSourceLink, so that 支持多对多关联。

#### Acceptance Criteria

1. THE System SHALL review existing SourceLink structure in KnowledgeNodeModels.swift
2. THE System SHALL review existing ExtractedSourceLink structure in KnowledgeAPIModels.swift
3. THE System SHALL support one dimension mapping to multiple days/data records
4. THE System SHALL support one day/data record mapping to multiple dimensions
5. THE Refactored_SourceLink SHALL contain: id, nodeId, sourceType, sourceId, dayId, snippet, relevanceScore, extractedAt

### Requirement 12: 与现有代码的差异分析

**User Story:** As a 开发者, I want 明确新设计与现有代码的差异, so that 能制定清晰的迁移计划。

#### Acceptance Criteria

1. THE Design_Document SHALL include a gap analysis comparing new design with existing code
2. THE Gap_Analysis SHALL cover: KnowledgeNodeModels.swift, AIPreferencesModels.swift, NarrativeProfileModels.swift, NarrativeRelationshipModels.swift
3. THE Gap_Analysis SHALL identify: 已完成项、未完成项、需废弃项、需更新项
4. THE Gap_Analysis SHALL provide migration recommendations

### Requirement 13: 维度数据序列化

**User Story:** As a 开发者, I want 维度数据能正确序列化和反序列化, so that 数据能持久化存储。

#### Acceptance Criteria

1. FOR ALL dimension nodes, serializing then deserializing SHALL produce an equivalent object
2. THE System SHALL use JSON format for dimension data storage
3. WHEN serializing AttributeValue, THE System SHALL preserve the type information
4. THE System SHALL handle optional fields gracefully during deserialization
5. THE System SHALL maintain backward compatibility with existing data format
