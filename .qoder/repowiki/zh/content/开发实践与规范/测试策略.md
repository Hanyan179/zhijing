## 测试目录结构

项目测试文件位于根目录的 `Tests/` 目录下，与主应用代码分离。这种结构设计是为了避免主应用目标（App target）在编译时包含 `XCTest` 框架，因为 `XCTest` 仅在测试目标中可用。

测试文件按功能模块进行组织，每个测试文件对应一个特定的功能或组件。例如：
- `AIConversationModelsTests.swift`：AI对话模型测试
- `MarkdownParserTests.swift`：Markdown解析器测试
- `RichTextRendererTests.swift`：富文本渲染器测试
- `SyntaxHighlighterTests.swift`：语法高亮测试
- `MessageBubbleTests.swift`：消息气泡UI组件测试

这种按功能模块划分的组织原则使得测试代码结构清晰，便于维护和定位。

## 单元测试编写规范

单元测试使用 `XCTest` 框架编写，主要针对数据模型、业务逻辑和工具类进行测试。以下以 `AIConversationModelsTests.swift` 为例，说明如何使用 `XCTest` 框架测试数据模型的序列化、反序列化和业务逻辑。

### AI对话模型测试

`AIConversationModelsTests.swift` 文件中的测试主要验证 `AIConversation` 和 `AIMessage` 模型的正确性。测试内容包括：

1. **序列化与反序列化**：验证 `AIConversation` 对象在经过 JSON 序列化和反序列化后，所有属性（如ID、标题、消息列表、关联日期等）都能正确保留。
2. **消息时间顺序**：验证 `sortedMessages` 属性能始终按时间戳升序返回消息。
3. **唯一性**：验证新创建的对话ID是唯一的。
4. **日期关联一致性**：验证当对话收到新日期的消息时，该日期会正确添加到 `associatedDays` 数组中，且不会产生重复。
5. **对话分组完整性**：验证对话按天分组的逻辑是正确的，即一个对话应出现在其所有关联日期的分组中。

这些测试通过生成随机数据并进行大量迭代来验证模型的属性，确保在各种情况下模型行为的正确性。

### 文本解析和渲染功能测试

以 `MarkdownParserTests.swift` 和 `RichTextRendererTests.swift` 为例，说明如何测试文本解析和渲染功能，包括边缘情况处理。

#### Markdown解析器测试

`MarkdownParserTests.swift` 文件中的测试验证了 `MarkdownParser` 的正确性。测试内容包括：

1. **解析的幂等性**：对于任何有效的Markdown字符串，多次解析应产生相同的AST（抽象语法树）结构。
2. **增量解析收敛性**：对完整Markdown字符串的增量解析结果应与完整解析结果一致。
3. **不完整语法检测**：验证 `hasIncompleteSyntax` 函数能正确识别不完整的语法，如未闭合的代码块（````）或内联代码（`）。
4. **流式解析行为**：模拟流式传输，验证在解析逐步增长的前缀时，解析行为是合理的。

这些测试通过生成随机的Markdown内容来覆盖各种语法元素，确保解析器在各种情况下都能正确工作。

`RichTextRendererTests.swift` 文件中的测试验证了 `RichTextRenderer` 组件的正确性。测试内容包括：

1. **内联格式嵌套**：验证粗体、斜体、内联代码等内联格式的嵌套（如粗体中的斜体）能被正确渲染。

### 语法高亮功能测试

以 `SyntaxHighlighterTests.swift` 为例，展示语法高亮功能的测试用例设计。

#### 语法高亮器测试

`SyntaxHighlighterTests.swift` 文件中的测试验证了 `SyntaxHighlighter` 的正确性。测试内容包括：

1. **语法高亮保持代码结构**：验证应用语法高亮后，代码的实际文本内容不会改变，只添加格式属性。
2. **幂等性**：验证对已高亮的文本再次应用高亮，结果应保持不变。
3. **空代码处理**：验证空代码字符串能正确处理并返回空的 `AttributedString`。
4. **空白字符保留**：验证空格、制表符和换行符等空白字符在高亮后能被正确保留。
5. **显示名称一致性**：验证 `displayName(for:)` 函数能为各种语言返回正确的显示名称（如 "swift" -> "Swift"）。
6. **代码复制精确性**：验证复制代码块时，复制到剪贴板的文本与原始代码完全一致，不包含语法高亮的标记。

这些测试确保了语法高亮功能在保持代码完整性的同时，提供良好的视觉体验。

## UI测试编写规范

UI测试主要验证用户界面组件的布局、交互和视觉表现。以 `MessageBubbleTests.swift` 为例，说明UI组件的布局和交互测试。

### 消息气泡测试

`MessageBubbleTests.swift` 文件中的测试验证了 `MessageBubble` 组件的交互功能。测试内容包括：

1. **复制消息功能**：验证用户可以复制AI消息的内容。测试中检查了消息内容不为空，但实际的剪贴板操作需要UI测试框架。
2. **复制思考过程**：验证当AI消息包含思考过程时，用户可以选择复制思考过程和最终回答。
3. **上下文菜单**：验证用户消息和AI消息的上下文菜单不同。用户消息只有“复制”选项，而AI消息有“复制”和“重新生成”选项。
4. **重新生成回调**：验证“重新生成”按钮的回调函数能被正确调用。
5. **消息操作视图可见性**：验证只有AI消息才显示操作按钮（复制和重新生成）。

这些测试通过模拟用户交互，确保UI组件的行为符合预期。

## 测试执行流程

测试可以在Xcode中运行，也可以通过脚本批量执行。

### 在Xcode中运行测试

推荐的测试执行方式是创建一个正式的测试目标（Test Target）：
1. 在Xcode中：File → New → Target → Unit Testing Bundle
2. 将 `Tests/` 目录下的所有测试文件添加到新的测试目标中
3. 使用快捷键 Cmd+U 运行测试

这种方式可以在Xcode中正常使用测试导航器查看测试结果，并集成CI/CD流程。

### 使用脚本批量执行

项目提供了 `run_all_tests.swift.script` 脚本，可以批量运行所有测试。此外，也可以使用Swift Package Manager来运行测试：
1. 创建 `Package.swift` 文件并配置测试目标
2. 运行 `swift test` 命令

## 测试覆盖率要求

所有新功能都必须配套单元测试。虽然当前项目由于测试文件与主代码分离，无法直接在Xcode中查看代码覆盖率报告，但建议未来改进为添加正式的测试目标。这样可以：
- 在Xcode中正常运行测试
- 使用测试导航器查看测试结果
- 集成CI/CD测试流程
- 查看代码覆盖率报告

通过这些措施，可以确保代码质量和功能的稳定性。
