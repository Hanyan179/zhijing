## 分支管理策略

本项目采用基于 `main` 主分支的 Git 工作流，所有新功能开发均从 `main` 分支创建独立的功能分支。核心分支包括 `main` 作为主干分支，用于发布稳定版本。功能开发需创建 `feature/[功能名称]` 格式的分支，如 `feature/ai-conversation-enhancement`。紧急修复使用 `fix/[问题描述]` 格式的修复分支。功能完成后，需将功能分支合并回 `main` 分支，并可选择删除已合并的分支。禁止直接在 `main` 分支上进行开发，确保主干代码的稳定性。

## 提交信息格式要求

提交信息需遵循语义化格式，包含类型标签和简短描述。格式为 `<类型>: <简短描述>`，可选添加详细说明和关联信息。类型标签包括：`feat`（新功能）、`fix`（Bug 修复）、`refactor`（重构）、`docs`（文档）、`test`（测试）、`style`（代码格式）、`perf`（性能优化）、`chore`（构建/工具）、`revert`（回滚）。例如："feat: 实现 AI 对话历史记录功能" 或 "fix: 修复时间轴崩溃问题"。提交前需确保代码能编译通过、测试通过，并更新相关文档。

## 代码审查流程

代码审查是合并前的必要步骤。在合并功能分支前，应使用 `git diff main..feature/xxx` 查看所有更改，确保没有意外的文件被修改。审查时需确认代码符合项目设计原则，如使用原生 SwiftUI 组件、避免硬编码字符串和魔法数字。同时检查是否更新了相关文档，并运行 `bash Docs/validate_docs.sh` 验证文档格式。审查通过后方可合并到 `main` 分支。

## 项目设计原则

### 优先使用原生SwiftUI组件
项目UI开发优先采用原生SwiftUI组件，确保与系统风格一致并获得最佳性能。自定义组件应封装在UI/Atoms、Molecules、Organisms目录中，遵循原子设计原则。

### 禁止硬编码字符串
所有用户可见的字符串必须通过 `Localization.tr()` 方法获取，禁止在代码中直接使用硬编码字符串。支持的语言包括简体中文、繁体中文、英语、日语、韩语等。例如，在 `ProfileScreen.swift` 中使用 `Localization.tr("profileTitle")` 而非直接字符串。

### 避免魔法数字
代码中禁止出现魔法数字，所有常量应定义在设计系统中。颜色常量定义在 `Colors.swift` 中，如 `Colors.indigo`；字体常量定义在 `Typography.swift` 中，如 `Typography.body`。这确保了设计一致性并便于全局调整。

## 数据格式标准

### 日期格式标准
统一使用 `yyyy.MM.dd` 格式表示日期，如 "2024.12.18"。此格式具有良好的可读性、排序友好性，并符合国际化标准。所有模型的日期字段均遵循此格式，包括 DailyTimeline、AIConversation、MindStateRecord 等模型的日期字段。

### 工具函数
使用 `DateUtilities` 工具类处理日期转换。`DateUtilities.today` 获取当前日期字符串，`DateUtilities.format()` 将 Date 对象转换为字符串，`DateUtilities.parse()` 将日期字符串解析为 Date 对象。日期字符串可直接比较和排序，因为统一格式保证了字符串排序即时间排序。

## 测试策略

### 单元测试
单元测试文件位于 `Tests/` 目录下，包括 `AIConversationModelsTests.swift`、`AISettingsTests.swift` 等。测试文件因Xcode项目配置原因被移出主源码目录，避免编译错误。建议创建正式的Test target来运行这些测试。测试内容涵盖模型序列化、消息排序、唯一ID生成等属性测试。

### UI测试
UI测试通过SwiftUI的View测试功能实现，验证界面组件的正确性和交互逻辑。测试用例覆盖了从原子组件到完整页面的各个层次。测试文件如 `MessageBubbleTests.swift` 验证消息气泡组件的渲染和交互。

### 测试运行
可通过创建Xcode Test target、使用Swift Package Manager或直接运行Swift脚本三种方式运行测试。推荐使用Xcode的测试功能，以便使用测试导航器和查看代码覆盖率。

## 文档验证脚本使用说明

### 脚本功能
`validate_docs.sh` 脚本用于验证文档格式一致性，检查所有Markdown文件是否包含导航链接、版本号、作者、更新日期和状态等元数据。脚本会跳过 `README.md` 和空文件。

### 使用方法
在项目根目录下运行：
```bash
bash Docs/validate_docs.sh
```
脚本将输出每个文件的验证结果，通过✅、失败❌或跳过⏭️标识。失败的文件会列出缺少的元数据项。

### 测试脚本
`DocumentFormatTests.swift.script` 是完整的属性测试套件，可通过 `swift` 命令直接运行，提供更全面的文档格式验证。
